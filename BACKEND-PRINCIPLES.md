# Backend Principles (Production System)

Правила разработки backend. Цель — архитектурная целостность, безопасность данных, масштабируемость.

---

## 1. База данных — источник истины

База защищает данные сама: FK constraints, CHECK constraints, NOT NULL где возможно, RESTRICT вместо CASCADE для исторических данных. Нельзя полагаться только на код приложения. История (движения склада, производство) никогда не удаляется.

## 2. Исторические данные неизменяемы

Операционные журналы работают по принципу append-only: StockMovement, ProductionLog, ProductionOrder. Новая запись = новое событие, старые записи не изменяются. Это гарантирует корректный аудит.

## 3. Бизнес-идентификатор != технический id

Каждая сущность имеет: id (технический ключ), code (бизнес-идентификатор), name (отображаемое название). Никогда не использовать id как артикул.

## 4. Количества и деньги

Float запрещён. Используется Decimal. quantity > 0, направление движения определяется type. Никаких отрицательных количеств для определения направления.

## 5. Склад — через движения

Остаток никогда не хранится напрямую. Остаток = SUM(stock_movements) с CASE по type. Любое изменение склада = новое движение.

## 6. Производство фиксируется заказами

Производство через ProductionOrder. Заказ содержит snapshot BOM. Изменение BOM не влияет на существующие заказы.

## 7. Любое критичное действие — транзакция

Операции, изменяющие несколько таблиц, выполняются только в транзакции. Пример: completeOrder = списание сырья + приход продукции + изменение статуса. Все операции атомарны.

## 8. Чёткая граница слоёв

Route -> Service -> Prisma/DB. Route: проверка запроса, auth. Service: бизнес-логика. Route не работает напрямую с Prisma.

## 9. Аутентификация разделена

Гибридная модель: Worker + PIN для терминала, User + email/password для web. Worker может быть связан с User.

## 10. Авторизация через роли

RBAC: ADMIN, DIRECTOR, WAREHOUSE, WORKER. Роли проверяются в middleware (централизованно в lib/auth.ts). Не разбрасывать проверки ролей по коду.

## 11. Аудит обязателен

Ключевые таблицы хранят: createdBy, updatedBy, createdAt, updatedAt. Необходимо для расследования ошибок и прозрачности действий.

## 12. Никакой бизнес-логики в названиях

Запрещено кодировать логику в строках (например "левый" в названии). Используются поля: side, type, status.

## 13. Миграции должны быть воспроизводимы

Любая среда поднимается через: docker compose up, prisma migrate deploy, prisma db seed. Без ручных действий.

## 14. Docker — стандарт развёртывания

Каждый инстанс: app + postgres + .env. Без ручной настройки серверов.

## 15. Проект развивается эволюционно

Новые функции: простая модель -> стабильность -> расширение. Не добавлять сложные системы (permissions, lot tracking, routing), пока их не требует бизнес.

---

## Главный принцип

Backend должен быть предсказуемым, безопасным и расширяемым. Любое изменение архитектуры должно сохранять эти свойства.
