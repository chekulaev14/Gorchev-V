# QA Principles

Правила тестирования и обеспечения качества. Цель — гарантировать целостность данных и корректность бизнес-процессов, а не только UI.

---

## 1. UI-успех не является доказательством

Тост "успешно" ≠ корректная операция. Proof — данные в БД: StockMovement, StockBalance, InventoryOperation, ProductionOrderItem, StatusHistory. Каждый критичный тест проверяет состояние данных, а не только ответ UI.

## 2. System invariants

Правила, которые всегда должны быть истинны:

- StockMovement append-only: существующие движения не изменяются и не удаляются
- StockBalance = сумма всех StockMovement по itemId + locationId
- Любая операция изменения остатков проходит через InventoryOperation
- Один operationKey = одна операция; повторный запрос возвращает существующий результат
- COMPLETED заказ нельзя завершить повторно (409)
- Завершение заказа использует snapshot BOM, не текущий
- Рекурсивная сборка атомарна (вся цепочка от сырья до изделия в одной транзакции)
- StockBalance может быть отрицательным (сырьё не оприходовано, но фактически на складе)
- BOM не содержит циклов (ни прямых, ни косвенных)
- Side-совместимость: LEFT output не содержит RIGHT inputs, RIGHT output не содержит LEFT inputs, NONE output содержит только NONE inputs. Смешивание LEFT+RIGHT в одном шаге/составе запрещено
- Worker auth не даёт доступ к web-модулям
- WAREHOUSE не имеет доступ к admin API
- Unauthenticated → protected API → 401

Нарушение любого инварианта — блокирующий баг.

## 3. Severity

S1 — повреждение данных, обход авторизации, частичная транзакция → блокирует релиз.
S2 — неправильный статус/snapshot/остаток, доступ к чужому модулю → блокирует если P0.
S3 — валидация, soft delete, UI flow → по согласованию.
S4 — косметика → backlog.

## 4. Error semantics

Бизнес-ошибки бросаются как ServiceError с корректным HTTP-статусом. handleRouteError маппит ServiceError → status + message. Всё остальное → 500. Если тест получает 500 на бизнес-кейс — это баг в коде, а не в тесте.

Правило: Error → 500 = баг. ServiceError → 4xx = ожидаемое поведение.

## 5. Тесты бьют по реальным route

Тесты должны вызывать API точно так, как это делает фронтенд. Несуществующий route — false positive, а не баг. Перед написанием теста — проверить route handler и формат запроса.

## 6. Идемпотентность проверяется на уровне данных

Double POST с одним operationKey → одна InventoryOperation, одно движение. Проверка: count движений до и после, баланс не изменился при повторе.

## 7. Тестовая среда изолирована

Тесты на prod БД запрещены для write-операций. Test DB — отдельная PostgreSQL, .env.test, ENV guard. Seed воспроизводим. После каждого прогона — состояние предсказуемо.

## 8. Regression policy

После любого фикса — прогон по затронутым областям:

- Inventory: supplier income, assembly (+/-), balance consistency
- BOM: cycle check, snapshot isolation, bom-constructor / routing-constructor
- Auth: worker login, user login, protected routes + API
- Production orders: create, complete, repeat guard, status history

## 9. Конкурентность — manual

Конкурентные сценарии (две сборки на один остаток, два завершения одного заказа) не автоматизируются. Проверяются вручную, результат фиксируется в отчёте.

## 10. Порядок тестирования после изменений

1. tsc --noEmit (компиляция)
2. lint
3. Smoke (health, login, основные страницы)
4. Затронутые бизнес-сценарии
5. Balance consistency (баланс = сумма движений)

## 11. Smoke чеклисты

Test smoke (после изменений):

- /api/health → 200
- Worker login по PIN → каталог
- User login по email/password → склад
- Основные страницы грузятся без ошибок
- Logout инвалидирует сессию
- WORKER не попадает на склад
- WAREHOUSE не попадает в управление пользователями
- Неверный PIN / пароль → отказ
- Unauthenticated → protected API → 401

Prod smoke (после деплоя, только read-only):

- /api/health → 200
- Login / logout работает
- Страницы доступны после login

На prod запрещено: приход, сборка, завершение заказов, любые write-операции.

## 12. Пока не автоматизируется

- UI bom-constructor / routing-constructor (стабилизируется)
- Конкурентные операции (manual)
- Быстро меняющиеся фичи

Правило: сначала стабилизировать manual, потом автоматизировать.

## Главный принцип

Тест доказывает корректность данных, а не UI. Если баланс сошёлся, операция идемпотентна, snapshot зафиксирован — система работает. Всё остальное — косметика.
