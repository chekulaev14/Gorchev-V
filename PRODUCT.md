## Стек

Next.js + TypeScript, Tailwind CSS + shadcn/ui, PostgreSQL + Prisma ORM.

## Модель

Multi-Level BOM + Material Transaction Ledger.

Производство описывается через:
- производственные маршруты (Routing) — источник истины для производственного преобразования
- версионированные спецификации (BOM) — источник истины для состава изделия (⚠️ не используется в production flow, UI скрыт — вся работа идёт через маршруты)
- ledger движений материалов (StockMovement)

## Граф производства

Каждый этап производства = новая номенклатура (полуфабрикат).

Базовый случай — линейная цепочка из одного сырья:

Сырьё А → З1 → З2 → З3 → Изделие

Каждый следующий компонент производится из предыдущего.

Иногда компонент делается не из предыдущего этапа, а из другого сырья. Например, З2 имеет собственный рецепт (Сырьё Б → З2). Это не исключение внутри основной цепочки, а отдельный маршрут для З2.

Один Routing = последовательность шагов (1..N). Каждый шаг может иметь несколько входов (сборка), но ровно один выход. Ветвления (один выход → несколько следующих шагов) запрещены.

Поведение системы:

Рабочий сделал 10 З1:
  Сырьё А  -X кг
  З1       +10 шт

Рабочий сделал 10 З2 (свой маршрут из Сырья Б):
  Сырьё Б  -X кг
  З2       +10 шт

Рабочий сделал 10 З3:
  З2       -10 шт
  З3       +10 шт

При списании система находит маршрут производства конкретного компонента (RoutingStep где outputItem = этот компонент) и списывает все входные материалы шага (RoutingStepInput[]) пропорционально outputQty.

## Рекурсия между маршрутами

Внутри одного Routing шаги последовательные (1..N), каждый шаг может иметь несколько входов.
Рекурсия возможна только между разными Routing.

Рекурсивное производство запускается только если не хватает остатка входного item.
Если остатка хватает — повторное производство не запускается.

Пример:
  Routing A: Сырьё → З1 → З2
  Routing B: З2 → З3 → Изделие
  produce(Изделие, 10):
    нужен З3 → нужен З2
    если З2 на складе достаточно → просто списать
    если З2 не хватает → достроить недостающее по Routing A

## Главный принцип

ERP не хранит остатки напрямую. ERP хранит только движения материалов.

StockMovement хранит физическое количество движения, всегда положительное.
Направление движения определяется парой fromLocationId → toLocationId.

Примеры:
  EXTERNAL → MAIN       = приход на склад (от поставщика)
  MAIN → PRODUCTION     = передача в производство (списание)
  PRODUCTION → MAIN     = приход из производства
  MAIN → EXTERNAL       = отгрузка

Остатки рассчитываются из StockMovement как разница входящих и исходящих движений для конкретной location:
  balance(item, location) = SUM(qty WHERE to=location) - SUM(qty WHERE from=location)

Остаток может быть отрицательным — штатная ситуация (сырьё фактически на складе, но ещё не оприходовано).
StockMovement является источником истины (source of truth) для состояния запасов системы.

## Единицы

Сырьё — кг. Заготовки и изделия — шт.

## Правила

- Один активный producing маршрут на номенклатуру (Routing где последний RoutingStep.outputItemId = item). Нескольких маршрутов у одной позиции не бывает.
- Один полуфабрикат может использоваться в разных изделиях.
- Нет партий материалов.
- Нет продаж и клиентов.
- Хотя бы один склад.

## Сущности

- Item — номенклатура (material, blank, product)
- Routing + RoutingStep + RoutingStepInput — производственный маршрут. Routing привязан к конечному изделию. RoutingStep хранит: outputItemId, outputQty, processId. Входные материалы — через RoutingStepInput[] (itemId, qty, sortOrder). Каждый шаг может иметь несколько входов (сборка). stepNo задаёт порядок шагов. Поддерживает не-1:1 операции
- Bom + BomLine — версионированная спецификация изделия. ⚠️ Код и API существуют, но функция скрыта из UI и не задействована в текущем production flow. Вся логика списания идёт через Routing
- BomEntry — legacy/runtime-слой. Не участвует в production flow. Не используется для расчёта списания
- ProductionOperation — факт выпуска (что произведено, сколько, на каком шаге маршрута). Одна операция = один факт производства. Привязана к InventoryOperation через 1:1 связь
- ProductionOperationWorker — участник операции (кто, сколько сделал, тариф, заработок). Одна операция может иметь несколько рабочих
- StockMovement — ledger движений материалов (append-only). Источник истины для остатков. Количество всегда положительное, направление определяется парой fromLocationId → toLocationId
- InventoryOperation — логическая бизнес-операция, которая создаёт несколько StockMovement (производство, корректировка, инвентаризация). Используется для идемпотентности и аудита
- StockBalance — кэш остатков, пересчитывается из StockMovement. Не является источником истины
- ProductionOrder — производственное задание. Опциональная сущность для планирования, не обязательна для фиксации производства
- Location — склад или производственная зона. Системные location: MAIN (склад), EXTERNAL (внешний мир), PRODUCTION (производство), ADJUSTMENT (корректировки), SCRAP (брак). Системные location нельзя удалить
- Worker — рабочий (PIN-авторизация на терминале производства)

Детали полей и типов — в schema.prisma (источник истины для структуры БД).

## Маршрут (цепочка производства)

Маршрут определяет последовательность изготовления заготовок из сырья до изделия.

Сырьё → Заготовка1 → Заготовка2 → Заготовка3 → Изделие

Каждый шаг маршрута (RoutingStep) задаёт преобразование:
  inputs[] → outputItem + outputQty

Каждый input = RoutingStepInput(itemId, qty, sortOrder).

Примеры:

Простой шаг (один вход):
  inputs=[СырьёА, 0.5кг] → outputItem=З1, outputQty=1шт

Сборка (несколько входов):
  inputs=[З1, 2шт; З2, 3шт] → outputItem=Изделие, outputQty=1шт

Не-1:1 операция:
  inputs=[З2, 3шт] → outputItem=Изделие, outputQty=2шт
  (из 3 заготовок получается 2 изделия)

Списание для каждого входа: round((input.qty * количество) / outputQty, 4), ROUND_HALF_UP.

## Зачем цепочки

1. Рабочий через терминал отмечает что произвёл за смену. Система сама списывает вход и приходует результат.
2. Видно на каких этапах сейчас находится металл (сколько сырья, сколько каких заготовок).
3. Можно посчитать сколько изделий можно сделать с учётом всех этапов.

## Рабочий фиксирует только результат

Рабочий выбирает номенклатуру (не операцию), вводит количество деталей (не кг).

Пример:
  "Сделал 10 шт З2"

ERP автоматически:
  1. находит producing шаг маршрута (RoutingStep где outputItem = З2)
  2. списывает входной материал
  3. приходует результат

Движения:
  З1  -10
  З2  +10

## Несколько рабочих

В одной операции может участвовать несколько рабочих. Каждый имеет своё количество и свой тариф.

Пример:
  операция: З2, 20 шт
  рабочий1 → 12 шт
  рабочий2 → 8 шт

Движения создаются один раз на общее количество (20 шт).
Начисления — по каждому рабочему отдельно.
Каждый добавляемый рабочий подтверждается своим PIN.

Инварианты:
- SUM(workers.quantity) = общее количество операции
- один рабочий не может быть добавлен дважды
- тариф фиксируется на момент операции, не пересчитывается задним числом

## Зарплата

Оплата сдельная. Тариф хранится для номенклатуры.
Начисление: количество * тариф = заработок за операцию.
Вся выработка и начисления хранятся в базе.

## Парные детали (side)

side: LEFT / RIGHT / NONE.

Сторона может появиться на любом этапе цепочки.
Если хотя бы один элемент цепочки имеет сторону — итоговое изделие тоже имеет сторону.
Смешивать LEFT и RIGHT в одной цепочке нельзя.
Валидация side при сохранении/активации маршрута и BOM:
- output LEFT  → inputs только LEFT или NONE
- output RIGHT → inputs только RIGHT или NONE
- output NONE  → inputs только NONE
- Смешивание LEFT и RIGHT в одном шаге/составе запрещено
