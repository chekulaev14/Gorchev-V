# Frontend Principles

Цель фронтенда — предсказуемость изменений. Добавление поля или изменение UI должно требовать изменения ограниченного и заранее известного набора мест. Frontend — тонкий слой представления, backend определяет бизнес-логику и правила системы.

---

## 1. UI не содержит бизнес-логики

Frontend: отображает данные, собирает действия пользователя, вызывает backend. Frontend не определяет правила системы. Расчёты (canAssemble, stock balance, production logic) выполняются на backend или в orchestration-слое, но не внутри UI компонентов.

## 2. Архитектура компонентов

UI состоит из трёх уровней:

- UI primitives — базовые элементы (Button, Input, Modal, Toast, SearchableSelect). Не знают о домене.
- Presentation components — отображение данных (ItemForm, BomTree, OperationForm, ItemCard). Получают данные через props, не загружают данные, не выполняют вычисления домена.
- Orchestrators — контейнеры экранов. Получают данные, управляют состоянием, выполняют derived вычисления, передают данные вниз.

Презентационные компоненты не принимают решений.

## 3. Derived state не вычисляется в UI

Вычисления (canAssemble, shortages, operation preview) выполняются в orchestrator, в hooks или на backend. Presentation components получают уже готовые данные.

## 4. Единая модель сущностей

Каждая сущность имеет один frontend-тип. Тип используется в UI, в состоянии, в отображении. Типы не дублируются в разных местах.

## 5. Поля описываются централизованно

UI-описание полей хранится в field config. Поле определяется один раз: key, label, type, visible, editable, formatter. Конфиг используется формами, карточками, таблицами. Добавление поля требует изменения: type + field config.

## 6. Форма сущности — один компонент

Каждая сущность имеет одну форму с режимами: create, edit, view. Запрещено создавать отдельные ItemAddForm, ItemEditForm, ItemCreateForm.

## 7. Повторяющийся UI становится примитивом

Если UI-паттерн используется в нескольких местах, он выносится в общий компонент (SearchableSelect, GroupedAccordion, ConfirmDialog, Toast). Копирование UI-логики запрещено.

## 8. Компоненты не работают с инфраструктурой напрямую

Presentation components не выполняют HTTP запросы, не используют инфраструктурные утилиты, не работают с backend напрямую. Всё взаимодействие через orchestration-слой.

## 9. UI состояние должно быть минимальным

Различать: server state, form state, ui state, derived state. Derived state не хранится, если может быть вычислен.

## 10. Ограничения архитектуры

Запрещено: бизнес-логика внутри JSX, вычисления доменных правил в presentation компонентах, дублирование типов, ручные реализации общих UI-паттернов, большие компоненты (>300 строк).

---

## Главный принцип

Frontend строится вокруг сущностей, единых UI-примитивов и предсказуемых точек изменения. Если изменение одного поля требует редактирования множества экранов — архитектура нарушена.
