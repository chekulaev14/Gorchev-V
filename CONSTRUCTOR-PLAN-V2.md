# Конструктор производственной цепочки V2

Отдельный внешний сервис, не встраивается в ERP. Standalone HTML на React Flow (@xyflow/react v12), раздаётся заказчику по ссылке. Заказчик сам строит производственную цепочку (номенклатура + маршруты) в визуальном конструкторе, экспортирует JSON — данные импортируются в ERP через API. Не заменяет BOM/Routing конструкторы внутри ERP, существует параллельно как инструмент для клиента.

## 0. Layout экрана — 3 зоны

```
┌──────────┬──────────────────────────┬──────────────┐
│  ЛЕВАЯ   │         ЦЕНТР            │   ПРАВАЯ     │
│ Дерево   │    Граф конструктора     │  Свойства    │
│ структуры│    (React Flow)          │  элемента    │
│          │                          │              │
│ Изделие  │  [steel]→[blank]→[prod]  │ Название     │
│  └─Загот │        ↑                 │ Тип (auto)   │
│    └─Лист│     [bolt]              │ Родители:    │
│          │                          │  blank ×2шт  │
│          │                          │  bolt  ×4шт  │
└──────────┴──────────────────────────┴──────────────┘
```

Левая панель — дерево структуры:
- Компактный обзор всей цепочки (когда граф большой)
- Клик по элементу → фокус на узле в центре + свойства справа
- Показывает иерархию: изделие → из чего → из чего
- Сворачивается кнопкой (toggle), центр занимает освободившееся место

Центр — граф конструктора (React Flow):
- Визуальная модель: сырьё слева → изделие справа
- Кнопки "+" для добавления узлов (у сырья "+" слева отсутствует — предыдущий этап невозможен)
- Автоматическое построение layout (dagre RL)
- Сборка: несколько компонентов сходятся в один узел
- Растягивается на всю ширину если боковые панели скрыты
- Zoom: масштабирование колёсиком мыши + кнопки +/- в углу графа. По умолчанию масштаб ~75% (отдалённый, чтобы вся цепочка была видна с запасом)

Правая панель — свойства выбранного элемента:
- Название (редактируемое)
- Тип (автоопределение, показывается но не вводится)
- Код/артикул (опционально)
- Список parents с qty ("сколько нужно для 1 единицы текущего?")
- Редактирование qty прямо в панели
- Сворачивается кнопкой (toggle), появляется при выборе узла

Обе боковые панели скрываемые — кнопка-стрелка на границе панели.

## 1. Контракт JSON v1

```json
{
  "schemaVersion": "1.0",
  "chain": {
    "name": "Кронштейн КН-200"
  },
  "items": [
    { "key": "item_1", "name": "Кронштейн КН-200", "type": "product", "unit": "pcs", "code": "" },
    { "key": "item_2", "name": "Заготовка КН-200", "type": "blank", "unit": "pcs", "code": "" },
    { "key": "item_3", "name": "Лист стальной 2мм", "type": "material", "unit": "kg", "code": "" }
  ],
  "steps": [
    {
      "key": "step_10",
      "stepNo": 10,
      "name": "Резка",
      "outputItemKey": "item_2",
      "outputQty": 1,
      "inputs": [{ "itemKey": "item_3", "qty": 0.8 }]
    },
    {
      "key": "step_20",
      "stepNo": 20,
      "name": "Финишная обработка",
      "outputItemKey": "item_1",
      "outputQty": 1,
      "inputs": [{ "itemKey": "item_2", "qty": 1 }]
    }
  ]
}
```

- finalItemKey не хранится — вычисляется: item без parents = final (product)
- item.type вычисляется автоматически по графу (см. раздел 3)
- item.unit вычисляется из type: material→kg, blank/product→pcs
- item.code — опциональный бизнес-ключ (артикул)
- qty хранится в parents[] каждого узла ("сколько нужно для 1 единицы текущего?")

## 2. Внутренняя модель UI

Не граф с отдельными edges. Каждый item хранит своих предшественников:

```
item = {
  key,           // автогенерация: item_1, item_2...
  name,          // вводит пользователь
  code,          // опционально
  parents: [     // "из чего делается"
    { itemKey, qty }
  ]
}
```

Edges и steps вычисляются динамически — не хранятся:

```
buildEdges(items)  → для React Flow визуализации
buildSteps(items)  → для JSON экспорта
```

Автоопределение type по позиции в графе:
- parents пусто → material (исходный компонент, ниоткуда не производится)
- есть parents + кто-то ссылается как parent → blank (промежуточный)
- есть parents + никто не ссылается как parent → product (конечное изделие)

Final item = item у которого есть parents, но его никто не указывает как parent.
Material тоже без parents, но он не final — у него нет входящих связей вообще.

Unit:
- blank/product → всегда pcs
- material → выбирается пользователем (dropdown: кг, м, м², л). По умолчанию кг

## 3. Правила валидации v1

| # | Правило | Код ошибки |
|---|---------|-----------|
| 1 | chain.name не пустой | CHAIN_NAME_REQUIRED |
| 2 | ровно один final item (никто не ссылается как parent) | MULTIPLE_FINAL_ITEMS |
| 3 | item.key уникален | ITEM_KEY_DUPLICATE |
| 4 | item.name не пустой | ITEM_NAME_REQUIRED |
| 5 | qty > 0 везде | INVALID_QTY |
| 6 | нет циклов в графе (проверять при добавлении parent, не при экспорте) | GRAPH_HAS_CYCLE |
| 7 | нет висячих items (не связаны с цепочкой) | GRAPH_HAS_ORPHAN |
| 8 | final item — output хотя бы одного step | FINAL_ITEM_NOT_PRODUCED |
| 9 | у одного item ровно один набор parents (один producing step) | ITEM_MULTIPLE_PRODUCERS |
| 10 | parent.itemKey существует | PARENT_NOT_FOUND |

Убраны правила про type/unit/edge direction — они вычисляются автоматически и ошибки невозможны.

## 4. Ключевой алгоритм: items → steps

```
Алгоритм buildSteps(items):

1. Для каждого item у которого parents.length > 0:
   - outputItemKey = этот item
   - outputQty = 1 (v1)
   - inputs = item.parents (развёрнутые: parent → input)
2. Topological sort по depth
3. Назначить stepNo: 10, 20, 30...
4. Назначить step.key: step_10, step_20...
```

Пример:

```
product (parents: [blank])
blank (parents: [steel, bolt])
steel (parents: [])
bolt (parents: [])

→ step_10: inputs=[steel, bolt] → output=blank
→ step_20: inputs=[blank] → output=product
```

## 5. Маппинг JSON → ERP сущности

```
items[]           → Item (name, type, unit, code)
chain             → Routing (productItemId = Item по finalItem)
steps[]           → RoutingStep (stepNo, outputItemId, outputQty, processId)
steps[].inputs[]  → RoutingStepInput (itemId, qty, sortOrder)
```

Pipeline импорта:
1. Принять JSON
2. Проверить структуру (schemaVersion, обязательные поля)
3. Проверить business rules (таблица выше)
4. Создать Item для каждого items[] → построить itemMap: { key → erpId }
5. Создать Routing: productItemId = itemMap[finalItem.key]
6. Создать RoutingStep для каждого steps[], используя itemMap для outputItemId
7. Создать RoutingStepInput для каждого inputs[], используя itemMap для itemId
8. Вернуть mapping: { key → erpId }

Всё в одной транзакции. Режим v1: только create.

## 6. Фронтенд-инварианты

Правила, которые действуют на всех фазах разработки:
- item — единственный источник истины. Edges, steps, types — производные
- edges никогда не редактируются напрямую — только через изменение parents
- каждое действие пользователя = доменная команда (createItem, addParent, removeItem, renameItem)
- блокировать действия, создающие цикл, на уровне UI — не только валидировать после факта
- отображать quantity на рёбрах графа
- предусмотреть undo/redo (архитектурно заложить историю команд)

## 7. Фазы реализации

### Фаза 0 — Статичный макет (утверждение визуала) ✅ DONE
- HTML макет 3 зон: левая панель, центр-граф, правая панель
- Заглушки узлов, стрелок, панелей — без логики, только внешний вид
- Стилистика: шрифты, цвета, размеры узлов, кнопки "+"
- Дизайн узла: название + иконка типа + unit (не просто текст)
- Состояния узлов: обычный, выделенный, с ошибкой (красная рамка)
- Цветовая кодировка типов: material (зелёный), blank (синий), product (фиолетовый)
- Мини-форма добавления: попап при клике на "+" (название + количество)
- Пустое состояние: ghost example (полупрозрачная демо-цепочка material → blank → product) + кнопка "Создать свою цепочку"
- Хедер: название цепочки (редактируемое), кнопки Демо / Очистить / Экспорт
- Цель: согласовать визуальный язык до написания кода
- Результат: утверждённый UI, без бизнес-логики, без React Flow

### Фаза 1 — Интерактивный каркас на React Flow ✅ DONE
- Реализация конструктора на финальном стеке (React Flow + dagre)
- RL direction: сырьё слева → изделие справа
- Демо-дерево с загрузкой данных
- Выбор узла → правая панель свойств
- Добавление узлов через "+"
- Автоматическое построение рёбер из item-структуры
- Quantity на рёбрах
- Базовые UI-ограничения против циклов
- Цель: первый рабочий сценарий на финальном стеке, без временных прототипов
- Результат: кликабельный каркас, который далее развивается без переписывания
- Не входит: валидация, генерация steps, экспорт, сохранение, undo/redo

### Фаза 2 — Guided constructor (items + parents)
- Внутренняя модель: items[] с parents[] (не отдельные edges)
- Edges и steps вычисляются динамически (buildEdges, buildSteps)
- Type и unit вычисляются автоматически по позиции в графе
- Guided constructor — кнопки "+" вместо рисования стрелок:
  - "+" слева = добавить предыдущий этап (линейное продолжение)
  - "+" сверху/снизу = добавить дополнительный вход (ответвление/сборка)
  - Все три "+" — одна операция: createItem() + добавить в parents текущего
  - Клик по "+" → мини-форма: название + количество (2 поля, больше ничего)
  - После сохранения: узел + связь + перестроение layout
  - Удаление узла = удалить item + убрать его из parents всех узлов (не каскадное удаление)
- Автогенерация key (item_1, item_2...)
- Auto-layout (dagre RL) пересчитывается при изменениях
- Undo/redo (базовая история команд)
- Tooltips на "+" при первом использовании ("добавить предыдущий этап" / "добавить компонент")

### Фаза 7 — Backend импорт

#### 7.1 API контракт
- POST /api/constructor/v1/validate — только проверка, ничего не пишет
- POST /api/constructor/v1/import — запись в БД, принимает importMode=createOnly
- dryRun=true для предпросмотра результата без записи
- schemaVersion обязателен в каждом запросе
- Формат ответа: { finalItem, itemsCreated, stepsCreated, mapping: { externalKey → erpId }, warnings[], errors[] }
- Error model: { code, message, path, itemKey, stepKey }

#### 7.2 Import pipeline
- constructor-import.service.ts — разбит на этапы:
  - parseTemplate() → validateSchema() → validateBusinessRules()
  - inferItemTypesAndUnits() → buildStepsFromItems() → buildImportPlan()
  - executeImport()
- validate — вне транзакции
- execute — внутри одной транзакции, при любой ошибке rollback всего
- Идемпотентность: importKey в запросе, повторный import с тем же ключом не создаёт дубли

#### 7.3 Правила импорта v1
- createOnly — только создание новых сущностей, без матчинга существующих items
- Создаётся: Item + Routing + RoutingStep + RoutingStepInput
- BOM не создаётся (только Routing)
- Routing создаётся в статусе DRAFT — внешний конструктор не влияет на production напрямую
- type/unit должны быть согласованы: product→pcs, blank→pcs, material→выбранная единица
- outputItem у step не может иметь второго producer
- Все stepNo уникальны после buildSteps
