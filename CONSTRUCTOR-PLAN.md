# План: эволюция конструктора

Цель: разделить конструктор на два экрана (BOM + Routing), добавить поддержку нескольких входов на шаг маршрута.

Контекст: данные в routing/routing_steps тестовые — миграция не нужна, делаем clean break. Старый UI конструктора полностью удаляется.

Ключевое изменение модели: RoutingStep переходит от "один вход → один выход" к "много входов → один выход" через новую таблицу RoutingStepInput.

---

## Фаза 1: БД

Что делаем:
- Очистить таблицы routing_steps, routings (тестовые данные)
- Убрать из RoutingStep поля: inputItemId, inputQty
- Добавить модель RoutingStepInput
- npx prisma migrate dev

RoutingStep после изменения:
- id, routingId, stepNo, processId
- outputItemId, outputQty
- normTimeMin, setupTimeMin, note
- inputs → RoutingStepInput[]
- @@unique([routingId, stepNo]) — два шага с одинаковым номером невозможны

RoutingStepInput (новая таблица):
- id
- stepId → RoutingStep (onDelete: Cascade)
- itemId → Item (onDelete: Restrict)
- qty: Decimal(10,4)
- sortOrder: Int (порядок входов в UI, стабильная сортировка)
- Без @@unique на [stepId, itemId] — один item может входить несколько раз по разным причинам
- @@unique([stepId, sortOrder]) — порядок входов уникален внутри шага
- @@index([stepId]) — быстрое чтение входов шага
- @@index([itemId]) — быстрый поиск "где используется позиция"

## Фаза 2: Backend — routing.service

Что меняем:
- createRouting() — steps принимают массив inputs вместо одного inputItemId
- updateRoutingSteps() — аналогично
- validateSteps() — правила:
  - каждый step имеет хотя бы один input
  - outputItemId уникален в маршруте (если бизнес потребует повтор — ослабить, но пока жёсткое правило)
  - stepNo непрерывные (1..N)
  - последний step.outputItemId = routing.itemId
  - qty > 0 для всех inputs и output
  - шаг не может использовать свой output как один из входов
  - шаг не может ссылаться на output, который появляется только в более позднем шаге
  - ровно один outputItemId на шаг, outputQty > 0 (явная проверка)
  - sortOrder уникален внутри одного шага (входы не "прыгают" при повторном открытии)
  - input.itemId не должен совпадать с outputItemId любого шага с stepNo >= текущего шага (защита от циклов внутри маршрута)
- getProducingStep() — возвращать step с массивом inputs
- getActiveRoutingByItem() — переименовать из ByProduct, include inputs
- getRoutingsByItem() — переименовать из ByProduct, include inputs
- activateRouting() — проверка что все inputs заполнены

API routes:
- POST /api/routing — новый формат payload (steps[].inputs[]). stepNo передаётся явно из UI, не вычисляется на сервере (иначе reorder непредсказуем)
- PUT /api/routing/[id] — новый формат, stepNo тоже явно
- GET /api/routing — возвращать inputs
- routing.schema.ts — обновить zod-схемы

## Фаза 3: Backend — production.service

Что меняем в produce():
- Получить producing step → массив inputs
- Рассчитать needed для каждого input: round((input.qty * quantity) / step.outputQty, 4), ROUND_HALF_UP. Зафиксировать в коде явно, не полагаться на дефолт JS
- visitedItems Set хранит пары (itemId, routingId) — защита от циклов. Очищать запись после завершения рекурсивной ветки, чтобы позиция могла быть произведена повторно в другой ветке

Порядок операций в produce() (строго):
1. Для каждого input: прочитать баланс (без lock), вычислить deficit = max(needed - available, 0). Если deficit > 0 → рекурсивно produce() для этого input на deficit
2. После завершения всех рекурсий: SELECT ... FOR UPDATE на балансы всех входов одним проходом, в порядке itemId ASC (защита от deadlock)
3. Повторно проверить available >= needed для каждого входа (баланс мог измениться). Если не хватает — ошибка "недостаточно входов после блокировки", транзакция откатывается
4. Списать все входы одновременно: StockMovement (from=MAIN, to=PRODUCTION) для каждого input
5. Создать output: StockMovement (from=PRODUCTION, to=MAIN)
6. Обновить StockBalance для каждого input + output
Не списывать частично. Блокировка только на шаге 2, не раньше.

## Фаза 4: UI — BOM-конструктор

Экран "Состав позиции":
- Левая панель: список позиций (изделия + заготовки) с поиском
- Основная область: дерево состава выбранной позиции
- Выбрал позицию → видишь из чего она состоит
- Добавить компонент → выбрать из номенклатуры, указать количество
- Удалить компонент
- Изменить количество
- Дерево многоуровневое: компонент может иметь свой состав (readonly, переход по клику)
- Сохранение через Bom + BomLine (версионирование: DRAFT → ACTIVE)

Использует: bom-version.service (уже есть)

BOM UI делается первым, потому что он проще и задаёт структуру изделия, от которой зависит routing.

## Фаза 5: UI — Routing-конструктор

Экран "Маршрут производства":
- Выбор позиции → показать её маршрут (шаги)
- Каждый шаг: список входов + один выход
- Простой случай (заготовка): один шаг, один вход (сырьё → заготовка)
- Сборка (изделие): один шаг, несколько входов (заготовка3 + заготовка4 → изделие)
- Длинная цепочка: несколько шагов последовательно

Формат: структурный редактор шагов (не canvas, не free-form). Шаги списком сверху вниз, у каждого шага: входы, выход, количества, параметры. Строгая форма с хорошей читаемостью.

- Добавить/удалить шаг (при удалении stepNo пересчитывается автоматически, без разрывов)
- В каждом шаге добавить/удалить входы
- Количества inline
- Сохранение через routing.service (DRAFT → ACTIVE)
- ACTIVE маршрут не редактируется вообще; любые изменения только через создание нового DRAFT

Бонус (не MVP, но полезно сразу): read-only блок "где используется эта позиция" — технолог видит, в какие BOM входит позиция, маршрут которой он редактирует.
