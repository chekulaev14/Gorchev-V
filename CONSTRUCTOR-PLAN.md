# План: Универсальный конструктор BOM

## Проблема

Текущий конструктор привязан к изделию. Нельзя задать рецепт заготовки (из какого сырья она делается) отдельно от цепочки изделия. Заготовки из одной колонки не связываются друг с другом, нет способа создать параллельные ветки.

## Решение

Один конструктор, корнем может быть любой элемент — и изделие, и заготовка. Слева — панель со списком всех элементов (табы: Все / Изделия / Заготовки, поиск). Выбрал элемент — справа открывается конструктор с ним как корнем.

- Выбрал заготовку → справа карточка заготовки, слева колонка сырья. Простая цепочка: сырьё → заготовка.
- Выбрал изделие → справа карточка изделия, слева заготовки с промежуточными колонками. Полная цепочка: сырьё → заготовки → изделие.

Рецепт каждого элемента сохраняется отдельно. У заготовок в списке видно "рецепт настроен" / "нет рецепта".

Превью: CONSTRUCTOR-PREVIEW.html

## Что меняется в БД

Ничего. BomEntry (parent→child, quantity) уже поддерживает оба уровня. Проверка циклов уже есть.

## Что меняется в backend

Ничего. bom.service уже работает с произвольными связями. API /api/bom — CRUD для BomEntry без ограничений по типам.

## Что меняется во frontend

### Шаг 1. Sidebar — панель выбора корня

Добавить левую панель в конструктор:
- Список элементов: изделия + заготовки
- Табы-фильтры: Все / Изделия / Заготовки
- Поиск по имени и артикулу
- Бейдж типа (Изделие / Заготовка) у каждого элемента
- Индикатор "рецепт настроен" у заготовок, у которых есть BOM-связи
- Клик по элементу — загружает его рецепт в конструктор

### Шаг 2. Универсальный корень

Убрать ограничение "корень = только изделие":
- Корневая карточка справа определяется выбранным элементом
- Если корень — заготовка: одна колонка сырья слева, без кнопки "+ Колонка"
- Если корень — изделие: колонка сырья + промежуточные колонки заготовок, кнопка "+ Заготовка"
- Связи, сохранение, потенциал — работают как раньше

### Шаг 3. Откат localWeights

Убрать localWeights из ChainConstructor. Вес заготовки задаётся в её рецепте (когда заготовка = корень), а в рецепте изделия показывается readonly.

## Порядок реализации

1. Шаг 1 — sidebar с выбором корня
2. Шаг 2 — универсальный корень (убрать привязку к изделию)
3. Шаг 3 — откат localWeights

## Что НЕ меняется

- БД, schema.prisma, миграции — не нужны
- bom.service, API routes — не нужны
- Потенциал, складские операции, производственные заказы — не затрагиваются
- BOM versioning (Bom + BomLine) — не затрагивается
