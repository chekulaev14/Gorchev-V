# Logging Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Добавить структурированное логирование (pino + requestId) во все API routes и ключевые сервисы ERP.

**Architecture:** Singleton pino-логгер в `lib/logger.ts`. Middleware добавляет `x-request-id` header (Edge Runtime). Wrapper `withRequestId()` инициализирует AsyncLocalStorage в Node.js runtime — requestId автоматически попадает в каждую строку лога без ручной передачи.

**Tech Stack:** pino, AsyncLocalStorage (Node.js built-in), Next.js App Router middleware, vitest (существующий тест-раннер).

**Spec:** `docs/superpowers/specs/2026-04-02-logging-design.md`

```typescript
// app/src/lib/logger.ts
import pino from 'pino'
import { AsyncLocalStorage } from 'async_hooks'

const als = new AsyncLocalStorage<string>()

const logger = pino({
  level: process.env.LOG_LEVEL ?? 'info',
  ...(process.env.NODE_ENV === 'development' && {
    transport: { target: 'pino-pretty', options: { colorize: true } },
  }),
})

export function getLogger() {
  return logger
}

export function runWithRequestId<T>(requestId: string, fn: () => Promise<T>): Promise<T> {
  return als.run(requestId, fn)
}

export function getRequestId(): string | undefined {
  return als.getStore()
}

// Логгер с автоматическим requestId в каждой строке
export const log = {
  info: (msg: string, data?: Record<string, unknown>) =>
    logger.info({ requestId: getRequestId(), ...data }, msg),
  warn: (msg: string, data?: Record<string, unknown>) =>
    logger.warn({ requestId: getRequestId(), ...data }, msg),
  error: (msg: string, data?: Record<string, unknown>) =>
    logger.error({ requestId: getRequestId(), ...data }, msg),
}
```

### Task 3: Обновить middleware.ts — добавить x-request-id

**Files:**
- Modify: `app/src/middleware.ts`

Существующий middleware: авторизация JWT + RBAC. Добавляем только генерацию UUID в заголовок. Не трогаем логику авторизации.

- [ ] **Step 1: Найти место вставки**

Открыть `app/src/middleware.ts`. Найти строку где создаётся `const response = NextResponse.next()` или финальный return. requestId нужно добавить ДО return.

- [ ] **Step 2: Добавить crypto.randomUUID() в middleware**

В начало функции `middleware()`, сразу после `const { pathname } = request.nextUrl`:

```typescript
const requestId = crypto.randomUUID()
```

Важно: requestId нужно прокинуть в **request headers** (не response), иначе route handlers не увидят его через `req.headers.get('x-request-id')`. При этом auth headers (x-actor-id, x-actor-role, x-worker-id) тоже переносим сюда — в один requestHeaders объект.

Заменить финальный блок (строки 35-43 в middleware.ts):

```typescript
// было:
const response = NextResponse.next();
response.headers.set("x-actor-id", auth.actorId);
response.headers.set("x-actor-role", auth.role);
if (auth.workerId) {
  response.headers.set("x-worker-id", auth.workerId);
}
return response;

// стало:
const requestId = crypto.randomUUID()
const requestHeaders = new Headers(request.headers)
requestHeaders.set('x-request-id', requestId)
requestHeaders.set('x-actor-id', auth.actorId)
requestHeaders.set('x-actor-role', auth.role)
if (auth.workerId) {
  requestHeaders.set('x-worker-id', auth.workerId)
}
return NextResponse.next({ request: { headers: requestHeaders } })
```

Все заголовки (auth + requestId) прокидываются в request headers одним вызовом.

Важно: `crypto.randomUUID()` доступен в Edge Runtime нативно, без импортов.

- [ ] **Step 3: Добавить withRequestId в logger.ts**

```typescript
// Добавить в конец app/src/lib/logger.ts
import type { NextRequest } from 'next/server'

type RouteHandler = (req: NextRequest) => Promise<Response>

export function withRequestId(handler: RouteHandler): RouteHandler {
  return async (req) => {
    const requestId = req.headers.get('x-request-id') ?? crypto.randomUUID()
    return runWithRequestId(requestId, () => handler(req))
  }
}
```

Примечание: тип `Promise<Response>` (базовый Web API Response) — то что ожидает Next.js App Router. `NextResponse` расширяет `Response`, поэтому обёртка совместима с любым route handler.
