# Маршруты 2: дизайн нового конструктора цепочек

Дата: 2026-04-03
Статус: согласованный дизайн, без реализации

## Цель

Добавить в ERP новый модуль `Маршруты 2` рядом с существующим разделом `Маршруты`.

Новый модуль должен:
- дать пользователю визуальный конструктор цепочки на базе `CONSTRUCTOR-PIPELINE-V2.html`
- не писать данные в БД во время редактирования
- по кнопке `Записать` одной командой создавать новую номенклатуру, если ее не было, и сохранять маршрут в существующие боевые таблицы ERP

Старый раздел `Маршруты` на первом этапе остается в системе как fallback и как второе представление тех же данных.

## Почему не заменяем сразу

Текущий routing UI уже работает и использует боевую модель `Routing`.
Новый конструктор дает другой UX и требует отдельного слоя публикации.
Чтобы не ломать рабочий сценарий, первый этап внедрения идет параллельно:
- старый `Маршруты` остается
- новый `Маршруты 2` создается рядом
- оба экрана работают с одной и той же моделью данных после публикации

После тестирования и стабилизации старый экран можно выводить из эксплуатации.

## Объем первой версии

Первая версия сознательно упрощена:
- одна цепочка в конструкторе публикуется как один `Routing`
- сложные графы с автоматическим разбиением на несколько `Routing` не поддерживаются
- существующую номенклатуру внутри конструктора редактировать нельзя
- draft persistence в БД не делаем
- после успешной записи показываем только сообщение `Сохранено`
- перехода в старый экран после сохранения нет

## Размещение в ERP

Новый раздел появляется рядом со старым в навигации склада:
- `Маршруты`
- `Маршруты 2`

Новый экран должен жить внутри существующего warehouse-модуля, а не как внешний HTML-файл.
`CONSTRUCTOR-PIPELINE-V2.html` используется как UX-референс и основа композиции, но реализация должна быть встроена как React-модуль проекта.

## UX-модель

Экран `Маршруты 2` состоит из трех зон:
- левая панель со структурой цепочки
- центральный canvas с узлами и связями
- правая панель свойств выбранного узла

Главное правило UX:
- пока пользователь редактирует цепочку, все живет только в состоянии конструктора
- ERP не получает ни номенклатуру, ни связи, ни маршрут до кнопки `Записать`

После успешной публикации:
- показать toast или статус `Сохранено`
- оставить пользователя на том же экране
- не делать редирект

## Типы узлов

Каждый узел конструктора может быть одним из двух вариантов:

### 1. Существующая номенклатура

Пользователь выбирает уже существующую позицию из ERP.

Правила:
- узел хранит ссылку на существующий `itemId`
- поля существующей позиции внутри конструктора не редактируются
- при публикации новый `Item` для такого узла не создается

### 2. Новая номенклатура

Пользователь создает новую позицию прямо в конструкторе.

Минимальный набор полей:
- `name`
- `type`
- `unit`
- `side`
- `pricePerUnit` — необязательное

Дополнительные правила:
- `code` пользователь не вводит
- `code` генерируется backend
- `category`, `description`, `images` в первую версию не входят

## Модель публикации

Новый экран не должен напрямую дергать старые CRUD-операции маршрутов по шагам из UI.
Нужна отдельная publish-команда, принимающая payload конструктора целиком.

Предлагаемый endpoint:
- `POST /api/routing-v2/publish`

Предлагаемый смысл:
- UI отправляет полную структуру конструктора
- backend валидирует и публикует ее в боевые сущности ERP

Это важно, потому что позже можно будет без слома UI вставить drafts:
- сейчас: `constructor payload -> publish -> Item/Routing`
- потом: `constructor payload -> draft`
- затем: `draft -> publish -> Item/Routing`

## Flow сохранения

При нажатии `Записать` backend делает одну транзакцию:

1. Валидирует payload конструктора
2. Проверяет, какие узлы ссылаются на существующие `Item`
3. Создает недостающие `Item` для новых узлов
4. Получает реальные `itemId` для всех узлов цепочки
5. Собирает один `Routing`
6. Создает `RoutingStep`
7. Создает `RoutingStepInput`
8. Возвращает результат публикации

Важное следствие:
- так как сохраняются обычные `Routing` и связанные сущности, новый маршрут автоматически будет виден и в старом разделе `Маршруты`
- дополнительной синхронизации между экранами не требуется

## Backend-граница

Публикация должна идти через отдельный service.
Route handler не должен работать с Prisma напрямую.

Предлагаемая ответственность:
- route: принять payload, провалидировать request schema, вызвать service, обернуть ошибки через `handleRouteError`
- service: вся логика публикации конструктора
- persistence: Prisma внутри service и транзакции

Это соответствует текущим правилам проекта:
- Prisma не вызывать из route handler
- multi-table операции делать через транзакцию
- ошибки route-level оборачивать через `handleRouteError`

## Связь с текущей моделью ERP

Первая версия не меняет боевую модель данных ERP.
Она лишь добавляет новый способ создания тех же сущностей:
- `Item`
- `Routing`
- `RoutingStep`
- `RoutingStepInput`

То есть `Маршруты 2` — это новый UI и новая publish-команда, а не новая доменная модель маршрутов.

## Ограничения и сознательно отложенные возможности

Не делаем в первой версии:
- хранение drafts в БД
- автосохранение на сервере
- редактирование существующей номенклатуры из конструктора
- автоматическое разбиение сложного графа на несколько `Routing`
- поддержку нескольких независимых производственных веток в одной публикации
- расширенный набор полей новой номенклатуры

Все это можно добавить позже без смены базового UX, если сохранить главную границу:
- UI работает с моделью конструктора
- backend отдельно публикует ее в боевые сущности ERP

## Риски первой версии

### 1. Несоответствие графового UI и линейной модели Routing

Конструктор визуально допускает более богатую форму графа, чем текущая ERP-модель `один Routing = одна последовательная цепочка`.
В первой версии это нужно ограничить валидацией и UX-правилами.

### 2. Неявные поля для новых узлов

Если в конструкторе не объяснить разницу между существующей и новой номенклатурой, пользователь может не понимать, почему один узел редактируется, а другой нет.
Это нужно явно показать в UI.

### 3. Будущее добавление drafts

Если UI начнет опираться на старые CRUD endpoint по шагам маршрута, переход на drafts станет дорогим.
Поэтому publish endpoint должен быть отдельным сразу.

## Рекомендуемая реализация

Технически брать за основу UX и композицию из `CONSTRUCTOR-PIPELINE-V2.html`, но не встраивать HTML как есть.
Реализацию делать в стиле текущего проекта:
- новый React-экран внутри warehouse-модуля
- orchestration на клиенте
- typed API через `api-client`
- отдельный backend service публикации

Это дает быстрый старт без архитектурного долга в критичных местах.
