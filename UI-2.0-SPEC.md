# UI 2.0 — Спецификация

Целевое видение ERP-интерфейса. Переход от page-centric к object-centric + action-driven подходу.
Прототип: erp-assistant-concept.html.

## Проблемы текущего UI (v1)

- Page-centric: навигация = список страниц, нет связного flow между экранами
- Read-heavy: основной UX — просмотр таблиц. Actions вторичны (базовый CRUD)
- Нет lifecycle: объекты не имеют явных статусов и переходов в UI
- Нет процессного контекста: пользователь не видит "откуда пришёл → что дальше"
- Модули изолированы визуально, но UI одинаковый для разных задач

## Целевая модель: object-centric + action-driven

Не BPM-система, не kanban-доска. Лёгкий ERP shell для малого масштаба (3-10 пользователей).

### Принципы

1. **Object = центр экрана.** Пользователь работает с объектом (заказ, сертификат, документ), а не со страницей. У объекта есть: данные, связи (граф), статус, действия.

2. **Actions первичны.** Не просто таблица, а набор явных действий на объекте: создать, провести, закрыть, сформировать. UI подсказывает следующий шаг.

3. **Lifecycle встроен.** У каждого объекта — простой набор статусов (не BPM). Статусы видны в списках, фильтруемы, определяют доступные actions.

4. **Контекст всегда виден.** Breadcrumb показывает не путь по меню, а процессный контекст: модуль → объект → состояние.

5. **Связи между объектами раскрыты.** Order → Documents, Document ← Certificates. UI показывает граф, а не плоский список.

6. **AI = исполнитель внутри процесса.** Не отдельный чат, а контекстный помощник: подбирает данные, формирует документы, находит отклонения. Знает текущий объект и его состояние.

### Sidebar

Модули, не страницы. Каждый модуль — изолированный бизнес-домен.

Текущие модули:
- Дашборд
- Склад (на паузе)
- Производство (на паузе)
- Документы (следующий в работе)

Будущие модули — только по реальной потребности клиентов. Финансы, Контракты — НЕ планируются.

### Global Search

Единый поиск по всем модулям: заказы, сертификаты, документы, номенклатура. Cmd+K.

### Breadcrumb

Процессный контекст, не путь по файловой системе.
Пример: Документы → Заказ #001 → Сертификат качества → Draft

### AI Panel

Контекстный ассистент. Знает текущий модуль, объект и его состояние.

Первичный интерфейс — 1-click actions (не чат):
- [Подобрать сертификаты] — для текущего документа
- [Проверить покрытие] — для текущего заказа
- [Собрать пакет] — сгенерировать все документы заказа

Actions меняются в зависимости от контекста (какой объект открыт, в каком статусе).

Чат — вторичный. Для нестандартных запросов: "найди сертификаты на 09Г2С за 2026 год", "покажи заказы без welding".

Не просто чат — исполнитель с доступом к API.

## Модуль Документы (первая реализация UI 2.0)

Три объекта, три таба, три роли:

### Orders (workspace, entry point)
- Инженер начинает здесь
- List: заказы с Customer, статус (active/completed)
- Object View: данные заказа, вложенные Documents, Coverage блок
- Primary CTA: [Собрать пакет по заказу] — главный use-case, генерация всех документов заказа
- Coverage: кликабельный. ✔ metal → показать какие сертификаты используются. ○ welding → предложить подобрать. Это action surface, не просто индикатор
- Actions: создать Document, собрать пакет, проверить покрытие

### Certificates (ingestion pipeline)
- Отдельный пул, не привязан к Order
- Pipeline: Uploaded → Parsed → Verified
- List: с фильтрами по поставщику, марке, стандарту
- Object View: данные OCR, PDF preview, supplier, OCR confidence (highlight полей с низкой уверенностью — чтобы verify не был слепым действием)
- Actions: Parse OCR → Verify
- Supplier: inline select + modal create

### Documents (assembly, secondary)
- Основной flow — через Order View
- Таб Documents — реестр для поиска по всем документам + quick create [+ Document] с выбором Order внутри
- Filter: Draft | Issued
- Object View: привязанные Certificates (M:N, usageType), Shipments, PDF preview
- Consistency check перед Issue: все ли usageType покрыты, нет ли конфликтов (разные стандарты). Inline warning, не блокирующий
- Actions: Generate → Issue → Clone (v+1)

## Модуль Склад + Производство (UI 2.0)

Тот же shell: sidebar | список с pipeline/фильтрами | карточка объекта с actions.
Три таба, три объекта — как в Документах:

### Номенклатура (каталог, аналог пула Сертификатов)
- Список позиций с фильтрами: тип (материал/заготовка/изделие), наличие маршрута
- Pipeline: по типу (Материалы | Заготовки | Изделия)
- Object View (карточка позиции): данные, маршрут (встроенный конструктор цепочки), текущий остаток, потенциал производства, история движений
- Actions: редактировать, создать маршрут, открыть конструктор

### Остатки (мониторинг, аналог workspace Заказов)
- KPI-карточки: всего позиций, ниже минимума, потенциал
- Pipeline: Дефицит | В норме | Избыток
- Список: позиция, остаток, мин.запас, статус
- Object View: история движений по позиции (откуда/куда/сколько/когда), график расхода
- Actions: создать приход, провести отгрузку, корректировка

### Производство (журнал операций)
- Pipeline: по периоду или по статусу заказа (Запланировано | В работе | Завершено)
- Список: дата, рабочий, что произвёл, сколько, заработок
- Object View (операция): детали операции, рабочие-участники, движения материалов, начисления
- KPI: операций за период, выпущено деталей, начислено ₽, активных рабочих
- Actions: (read-only, ввод через терминал)

### Терминал рабочего (отдельный модуль в sidebar)
- Не master-detail. Мобильный layout: крупные кнопки, PIN-авторизация
- Отдельный UI, не вписывается в shell — и не нужно. Это инструмент рабочего, не менеджера

### Конструктор маршрутов
- Встроен в карточку номенклатуры (Object View), не отдельный таб
- Визуальный редактор цепочки: Сырьё → Заготовка1 → ... → Изделие
- Каждый шаг: входы (несколько), выход, количество, процесс
- Активация/архивирование маршрута — прямо из карточки

## Принципы реализации

1. **Один модуль за раз.** Начинаем с Документов. Склад адаптируем позже.
2. **Общие компоненты.** Pipeline, Object View, Coverage — переиспользуемые примитивы. При реализации Документов создаём их как generic, чтобы Склад мог использовать.
3. **Backend определяет lifecycle.** Статусы, transitions, доступные actions — всё с бэкенда. UI только отображает и вызывает.
4. **Incremental migration.** Не переписываем всё сразу. Новый shell применяется к новым модулям, старые адаптируются по мере работы.
5. **UI-стек.** React + Tailwind CSS + shadcn/ui. Новые компоненты строятся на shadcn/ui примитивах. Кастомные UI-элементы — только если shadcn не покрывает кейс.

## Чего НЕ делаем

- BPM-движок, BPMN-диаграммы, конструктор процессов
- Approval chains, цепочки согласований
- Модули Финансы, Контракты, HR
- Полноценный kanban (pipeline = лёгкий фильтр по статусам)
- Workflow designer / no-code конструктор
